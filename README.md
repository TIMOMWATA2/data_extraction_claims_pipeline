Ginja AI - Healthcare Document Extraction Pipeline

An end-to-end data extraction pipeline built for the Ginja AI AI/ML Engineering Internship case study. This system processes mixed-format healthcare claims (digital PDFs, scanned images) and outputs structured, validated JSON/CSV records using a combination of traditional parsing and Vision Language Models (VLMs).

**🗂️ Repository Structure(inside Ginja_AI_Case_Study_Claims folder)**

src/extraction_pipeline.ipynb: The main Google Colab notebook containing the complete execution pipeline.

data/sample_inputs/: Contains the 6 sample healthcare claims (mixed PDF and JPG) used for testing.

data/outputs/: Contains the final structured csv and json outputs generated by the pipeline.

Writeup_Timothy_Amwata.pdf: The detailed 4-page report comparing OCR vs. VLM approaches and detailing architectural trade-offs.

**🚀 How to Run the Pipeline**

This pipeline was built and tested in Google Colab to leverage cloud compute for OCR and API processing.

Open the Notebook: Upload src/extraction_pipeline.ipynb to Google Colab, or open it directly via GitHub integration.

Set up API Keys: * Navigate to the Secrets tab (key icon) on the left sidebar in Colab.

Add a new secret named Groq_API_KEY and paste your active Groq API key.

Toggle the button to grant notebook access to the secret.

Data Ingestion: The notebook is currently set up to use the "GitHub Clone" mechanism. Running the second cell will automatically clone this repository into the Colab environment and route the sample_inputs folder to the pipeline. (Alternatively, uncomment the manual upload block to test your own private documents).
I personally initially used the first commented method of mounting my personal google drive and accessing the documents from a folder saved in MY DRIVE.

Run All: Click Runtime -> Run All. The pipeline will execute format detection, PDF/OCR extraction, VLM testing, normalization, validation, and export.

**The Output**: After running the code successfully, check the folder icon on left side menu of Colab for the JSON file and the CSV file.

**🛠️ Library Choices & Rationale**

pdfplumber: Chosen for Text-based PDFs. It is vastly superior to standard PyPDF2 because it retains spatial layout information, which is critical when parsing invoices with right-aligned amounts and tables.

pytesseract (Tesseract OCR): Chosen for image-based extraction. It is the industry-standard open-source OCR engine. While it requires careful pre-processing (OpenCV), it runs locally with zero latency or API costs, making it a highly efficient first-pass filter.

cv2 (OpenCV): Used to pre-process images before OCR. Applying grayscale conversion and OTSU thresholding binarizes the image, stripping out colored stamps and shadow noise that would otherwise confuse Tesseract.

groq (VLM API): Used for semantic extraction. By dynamically querying for the latest Llama/Qwen multimodal models, we bypass the brittleness of RegEx parsing and leverage the model's spatial and semantic reasoning to map complex layouts directly to JSON.

⚠️ Known Limitations & Edge Cases

RegEx Brittleness (OCR): The current parsing logic for OCR output relies on strict Regular Expressions (e.g., looking for the exact string "Member No:"). If a provider submits a valid claim but uses the phrase "Patient ID:", the OCR pipeline will fail to map the field and flag it as missing.

Handwriting: Tesseract struggles significantly with cursive handwriting. While our pipeline successfully flags missing fields caused by OCR failure, it cannot currently interpret complex physician handwriting without routing to the VLM fallback.

Multi-Page Invoices: The VLM Base64 encoder is currently hardcoded to process only page=1 of a document to save tokens and prevent API timeouts. Multi-page discharge summaries require document-splitting logic not implemented in this MVP.
